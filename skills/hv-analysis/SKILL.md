# 橫縱分析法深度研究

你正在執行一次橫縱分析法研究。這個方法的核心不是蒐集很多資料，而是把資料整理成兩條能互相交叉驗證的軸線：

- **縱軸**：研究對象怎麼一路走到今天
- **橫軸**：研究對象在今天和誰相比、差在哪裡

最後要把兩條軸線交會成一個清楚判斷，而不是停在資料堆疊。

## 什麼時候用

適合用在：

- 研究一個產品、公司、技術、概念或人物
- 想弄清楚某個東西是怎麼出現、怎麼演變、現在處在什麼位置
- 需要競品比較、發展脈絡、關鍵轉折、未來推演
- 最後要交付一份完整的研究報告，而不是短摘要

不適合用在：

- 單純名詞解釋
- 很短的整理摘要
- 單純改寫成文章
- 社群貼文、短文案、標題生成

如果目標是把素材寫成一篇可讀性高的長文，而不是做系統性研究，改用 `Article-writer`。

## 核心原則

### 1. 先追時間，再看位置

不要一開始就急著做競品表。先把研究對象的時間脈絡拉出來：

- 它是怎麼出現的
- 起點解的是什麼問題
- 哪些事件改變了它的方向
- 有哪些決策一路影響到今天

只有先把縱軸看清楚，橫軸比較才不會浮在表面。

### 2. 橫軸不是比功能表

橫向分析不是把功能、價格、參數列成表格而已。真正要比較的是：

- 路線差異
- 適用情境
- 使用者為什麼會選它
- 它在整個版圖中的位置

### 3. 最終產出一定要有判斷

報告不能只回答「發生了什麼」，還要回答：

- 這些歷史是怎麼塑造現在的
- 它的優勢從哪裡長出來
- 它的弱點是不是早期決策留下來的
- 接下來最可能怎麼走

## 工作流程

## 第一步：定義研究對象

先確認四件事：

1. 研究對象是什麼
2. 它屬於產品、公司、概念、技術或人物哪一類
3. 為什麼現在要研究它
4. 有沒有特別想看的面向

如果使用者講得已經很明確，就不要為問而問，直接開始。

## 第二步：連網蒐集資料

這個 skill 必須連網查資料，不能只靠既有記憶。

資料蒐集至少分三塊：

- **縱向資料**：起源、創辦團隊、關鍵時間點、產品或策略轉折、危機、轉型、里程碑
- **橫向資料**：競品有哪些、彼此差在哪裡、口碑如何、市場怎麼看
- **背景資料**：產業環境、關鍵論文、社群討論、使用者真實回饋

### 資料來源優先順序

優先順序如下：

1. 官方資料
2. 原始訪談、GitHub、產品文件、發表內容
3. 權威媒體原創報導
4. 社群討論與使用者回饋

如果研究的是學術概念、模型或技術路線，**一定要查 arXiv 或論文來源**。

### 蒐集完成後要先做自檢

問自己：

- 我能不能講出一個完整故事
- 我有沒有漏掉關鍵競品
- 核心判斷是不是只有單一來源支撐

如果答案不穩，就先補資料，不要急著寫。

## 第三步：做縱向分析

縱向分析是報告主體。

至少要回答：

- 它為什麼會出現
- 是誰把它做出來的
- 最初的定位和今天有什麼不同
- 哪些時間點是關鍵轉折
- 哪些早期決策一路影響到今天

不要寫成年表流水帳。要讓讀者看見：

- 事件之間有因果
- 轉折背後有理由
- 不同階段有不同矛盾

如果對象夠複雜，建議自然切成幾個階段，例如：

- 萌芽期
- 快速成長期
- 重估或轉型期
- 穩定擴張期

## 第四步：做橫向分析

先判斷競品情境：

### 情境 A：幾乎沒有直接競品

重點改成分析：

- 為什麼現在沒有直接競品
- 真正的替代方案是什麼
- 未來競爭最可能從哪個方向冒出來

### 情境 B：競品很少

就把每個主要對手拆開來講深一點。

### 情境 C：競品很多

選 3 到 5 個最具代表性的比較，其餘簡要帶過。

### 橫向分析要看的面向

- 技術或方法路線
- 產品形態
- 商業模式
- 目標使用者
- 核心優勢與短板
- 真實口碑
- 生態位與市場位置

不要只比規格。要寫出「它現在實際活成什麼樣子」。

## 第五步：做橫縱交會

這是整份報告最重要的部分。

至少要回答：

1. 哪些歷史決策造成今天的位置
2. 今天的優勢能追到哪個起點
3. 今天的劣勢是不是過去的副作用
4. 主要競品如果也放回時間軸，路線差在哪裡
5. 接下來最可能、最危險、最樂觀的劇本是什麼

如果這一段只是前面內容的摘要，代表你還沒真的分析到位。

## 寫作要求

### 語言

所有輸出一律使用**台灣繁體中文**。

包含：

- 研究報告正文
- 標題與小標
- 表格欄位
- 圖說與補充說明
- PDF 封面與作者資訊

### 文風

報告不是冷冰冰的顧問簡報，也不是口語過頭的社群文章。

建議風格：

- 有節奏
- 有敘事
- 有判斷
- 有細節
- 不裝熟

可以參考 `Article-writer` 的長文寫作原則，但要保留研究報告該有的結構清晰與資訊密度。

### 避免事項

- 套話
- 空泛形容詞
- 沒有來源支撐的結論
- 把不知道的事寫得像知道
- 為了好看亂補案例
- 中國網路慣用語與中國平台語境用字

## 產出格式

建議至少產出兩個檔案：

- `[研究對象]_橫縱分析報告.md`
- `[研究對象]_橫縱分析報告.pdf`

## Markdown 結構建議

```text
# 標題
> 研究時間：... | 所屬領域：... | 研究對象類型：...

## 一句話定義

## 縱向分析：從誕生到現在

## 橫向分析：今天它跟誰比、差在哪裡

## 橫縱交會洞察

## 資料來源

## 方法說明
```

## PDF 轉換

本 skill 內建 `scripts/md_to_pdf.py`，可把 Markdown 轉成排版完整的 PDF。

範例：

```bash
python [skill目錄]/scripts/md_to_pdf.py input.md output.pdf --title "研究對象名稱" --author "Pigo"
```

如果沒有指定作者，腳本會使用預設作者名。

## 品質檢查清單

交付前確認：

- [ ] 縱軸不是流水帳
- [ ] 橫軸不是規格表翻譯
- [ ] 關鍵判斷有來源支撐
- [ ] 有回答為什麼，不只回答發生了什麼
- [ ] 最終有明確洞察，不是摘要
- [ ] 全文為台灣繁體中文
- [ ] 沒有殘留舊名稱或舊語境
